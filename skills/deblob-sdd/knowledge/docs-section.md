# The Docs section — the plan, upfront

Docs is a forcing function: think through the documentation implications, plan
them, track which living docs will need updating — so docs don't get silently
dropped (the corner every project cuts under pressure).

- **The Docs section is the doc _plan_, not the docs.** The strategy for the
  living documentation; the living docs are written _from_ it, on the
  living-docs axis ([three-axes](three-axes.md)). The spec plans the docs; the
  README IS the doc.
- **Upfront homework, not a trailing nicety.** Answered at spec time — _how is
  this taught? which existing living docs are affected, from the get-go?_ — and
  reviewed at the spec gate, before implementation runs. Last in the quintet is
  reading order, not scheduling.
- Why so hard a line: observed compliance without enforcement is **zero** — it's
  the most glossed-over section, which is exactly why it's a hard spec
  requirement. Homework before play.
